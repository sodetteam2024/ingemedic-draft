Acceso al sistema
Roles de usuario
El sistema tiene 3 perfiles con diferentes niveles de acceso. El rol determina qué módulos puede ver el usuario y qué acciones puede ejecutar.
👩‍💼
Administrador
Acceso total a todos los módulos. Puede crear, editar, aprobar y configurar cualquier entidad del sistema. Es el único que ve la Bitácora y la Configuración.
🚚
Repartidor
Solo ve sus entregas asignadas. Puede avanzar el estado de una entrega y crear solicitudes de novedad sobre un equipo. No accede a inventario, clientes ni órdenes.
🏥
Cliente
Acceso al portal externo (Fase 2). Puede consultar sus equipos activos y solicitar nuevos servicios. No accede al sistema operativo interno.
ℹ️
Para Alejandro — diseño de login
El login es una pantalla única para todos los roles. El sistema detecta el rol automáticamente y redirige al área correspondiente. El administrador va al Dashboard. El repartidor va directamente a sus Entregas.

🎨 Para Alejandro — regla de color
El rojo (#C0392B) SOLO se usa para alertas críticas — equipos con falla, entregas con retraso, saldos pendientes. Nunca como color decorativo. El cian es el color de "todo va bien y está activo". El verde es "completado". Esta semántica es la que hace que el personal lea el sistema sin leer el texto.

Cómo funciona el sistema
Flujo principal del sistema
El flujo operativo de Ingemedic siempre empieza por tener el equipo registrado y el cliente creado. Sin esos dos elementos no se puede generar nada más.
1
📦 Inventario
→
Registrar el equipo biomédico con su serial, categoría y estado inicial Disponible
2
👤 Clientes
→
Registrar el cliente (IPS, persona natural o empresa) con sus datos de contacto y dirección
3
📋 Órdenes de servicio
→
Crear la orden vinculando cliente + equipo + repartidor. El sistema genera los documentos legales automáticamente
4
🚚 Entregas
→
El repartidor recibe la asignación, sale a campo y registra la entrega con el nombre de quien recibe
5
⚠️ Solicitudes
→
Si el equipo presenta fallas, el repartidor crea un ticket de novedad. El admin decide si va a mantenimiento o se da de baja
6
🛠️ Mantenimientos
→
El equipo entra al taller. Al cerrar el mantenimiento vuelve a Disponible o se registra como Baja
7
📊 Servicios prestados
→
El admin consulta cuántos días lleva activo cada equipo por cliente para calcular su facturación
⚠️
Dependencias críticas — leer antes de diseñar
No se puede crear una Orden de servicio sin tener primero un Cliente y un Equipo disponible en inventario. No se puede crear una Entrega sin una Orden de servicio activa. Estos flujos deben reflejarse en el diseño — los formularios cargan datos de los módulos anteriores.
Módulo 1
Inventario de equipos
Es la base de todo el sistema. Aquí viven todos los equipos biomédicos de Ingemedic con su estado en tiempo real.
✅
Este módulo no depende de ningún otro
Es el primero que se debe poblar antes de usar el resto del sistema. Sin equipos registrados no se pueden crear órdenes ni entregas.
📋
Lista de equipos con filtros
Tabla con todos los equipos. Filtros por estado (Disponible, En préstamo, En mantenimiento, Con novedad, Baja) y por categoría. Búsqueda en tiempo real por serial, marca, modelo o cliente. El estado se muestra con un badge LED parpadeante.
➕
Registrar nuevo equipo
Modal con: categoría, marca, modelo, serial, INVIMA, fecha de ingreso y observaciones. El código (EQ-0001) se genera automáticamente. El estado inicial siempre es "Disponible".
📂
Ficha del equipo (panel lateral)
Al hacer clic en un equipo se abre un panel lateral derecho con todos sus datos y su Línea de Vida — historial cronológico de todo lo que ha pasado con ese equipo (ingresos, alquileres, devoluciones, mantenimientos, alertas). Los eventos más recientes aparecen arriba.
⇄
Cambiar estado del equipo
Desde la ficha o desde el botón contextual de la tabla. Requiere ingresar un motivo obligatorio que queda registrado en la línea de vida. El botón de acción cambia según el estado: "Rentar" si está disponible, "Recibir" si está en préstamo, "Ver mantenimiento" si está en taller.
🎨 Para Alejandro — estados con LED
Cada estado del equipo tiene un badge con un punto de color que parpadea. Verde parpadeante = Disponible (listo para alquilar). Cian estático = En préstamo. Amarillo parpadeante lento = En mantenimiento. Rojo parpadeante rápido = Con novedad (urgente). Esta semántica visual reemplaza la lectura del texto.

Módulo 2
Clientes
Directorio de todos los clientes con sus equipos activos y estado de cuenta. Junto con el inventario, es prerequisito para crear órdenes de servicio.
✅
Este módulo no depende de ningún otro
Se puede poblar en paralelo con el inventario. Cuando ya hay clientes y equipos, se pueden crear órdenes de servicio.
🗂️
Directorio con tabs por tipo
Vista de lista con tabs: Todos, Pacientes (Persona natural), Clínicas/IPS, Empresas. Cada tarjeta de cliente muestra nombre, documento, zona, teléfono, equipos activos en custodia y estado de pago (Al día / Saldo pendiente).
💰
Estado financiero visual
Cada cliente muestra un badge de estado de pago. Las tarjetas con saldo pendiente tienen un borde izquierdo rojo para que el administrador las identifique antes de programar un nuevo servicio.
📋
Expediente del cliente (panel lateral)
Panel lateral con datos completos, lista de equipos en custodia y Línea de Vida del cliente — historial unificado de contratos, entregas, pagos y alertas ordenado de más reciente a más antiguo. Cada evento tiene un badge de color (ENTREGA, PAGO, CONTRATO, SALDO, ALERTA).
⚡
Acciones rápidas desde la ficha
3 botones en la parte superior del panel: Rentar equipo (lleva a nueva orden con el cliente precargado), Programar retiro (genera orden de devolución) y Registrar pago (registra pago al expediente).
🎨 Para Alejandro — la ficha como expediente médico
La ficha del cliente debe verse como un expediente clínico. El administrador la abre mientras habla por teléfono con el cliente y en segundos sabe: qué equipos tiene, si debe dinero y cuándo fue el último movimiento. La línea de tiempo vertical es el recurso principal — debe ser lo más legible posible con tipografía clara y badges de color bien diferenciados.
Módulo 3
Órdenes de servicio
El corazón operativo del sistema. Cada alquiler de equipo se gestiona a través de una orden de servicio que tiene su propio flujo de estados y genera los documentos legales automáticamente.
⚠️
Requiere: Clientes + Inventario
Para crear una orden se necesita seleccionar un cliente existente y un equipo disponible en inventario. Si no hay clientes o equipos registrados, no se puede crear una orden.

📄
Generación automática de documentos
El sistema genera los documentos legales según el tipo de orden. Para arriendo de O₂ domiciliario: Contrato, Acta de entrega, Declaración de capacitación, Pagaré. Para arrendamiento institucional: Contrato IPS, Acta de entrega IPS. Los documentos se generan al vuelo — no se almacenan.
✍️
Registro de firma
Cuando el repartidor completa la entrega, registra el nombre de quien recibió el equipo. La orden muestra si fue firmada o si la firma está pendiente.
⚡
Botón de acción dinámico
El botón principal cambia según el estado: "Aprobar O.S." → "Despachar a ruta" → "Registrar entrega" → "Cerrar y facturar". El operador siempre sabe qué hacer sin leer instrucciones.
🎨 Para Alejandro — lista con timelines horizontales
La vista principal de órdenes NO es una tabla tradicional. Cada fila muestra el timeline horizontal completo de la orden, para que el admin vea el estado de todas las órdenes de un vistazo. Las órdenes con alerta tienen borde izquierdo rojo y el nodo problemático parpadea.
Módulo 4
Entregas
Tablero de control de logística del día. El administrador ve en tiempo real qué está haciendo cada repartidor en campo.
⚠️
Requiere: Órdenes de servicio activas
Las entregas se crean automáticamente cuando una orden pasa a estado "Despachada". El repartidor no crea entregas — las recibe asignadas.
🗺️
Lista de rutas agrupada por repartidor
No es un Kanban de columnas. Es una lista donde cada repartidor tiene su propio bloque con todas sus asignaciones del día. Cada entrega muestra el timeline horizontal (Programado → En ruta → Entregado) con las horas reales.
🚨
Alertas de retraso
Si una entrega supera el tiempo estimado, la fila completa se tiñe de rojo con un mensaje "¡Retraso detectado!". El nodo del timeline correspondiente parpadea en rojo y se sacude levemente.
↩️
Entregas vs Retiros
Las entregas (equipo va al cliente) tienen badge cian con flecha ↗. Los retiros (equipo vuelve de donde el cliente) tienen badge púrpura con flecha ↩. La diferencia visual evita errores operativos.
🎨 Para Alejandro — vista del repartidor en móvil
En Fase 2 el repartidor tendrá su propia vista móvil. El diseño debe contemplar pantallas de 375px con botones grandes y legibles bajo el sol. El flujo del repartidor es simple: ver mis entregas del día → iniciar → completar. Tres pantallas máximo.
Módulo 5
Mantenimientos
Registro y seguimiento de todos los mantenimientos de equipos. Puede ser preventivo (programado) o correctivo (falla detectada).
⚠️
Requiere: Inventario (equipo existente)
Un mantenimiento siempre está vinculado a un equipo específico del inventario. Al abrir el mantenimiento, el equipo cambia automáticamente a estado "En mantenimiento".
🔴
Diferenciación Preventivo vs Correctivo
Cada fila tiene una franja de color en el borde izquierdo: cian = Preventivo (rutina programada), rojo = Correctivo (falla). Los correctivos activos tienen una animación de pulso sutil para llamar la atención.
🩺
Hoja de vida clínica del equipo
Panel lateral con historial de estados del mantenimiento (Abierto → En proceso → Cerrado), observaciones del cliente, actividades realizadas y enlace al reporte técnico PDF.
✓
Cierre de mantenimiento
Al cerrar, el admin registra las actividades realizadas, el técnico responsable (opcional) y decide si el equipo vuelve a "Disponible" o se da de "Baja". Esta acción actualiza el inventario automáticamente.
🎨 Para Alejandro — botón predictivo
El botón de acción en cada fila cambia según el estado: "🚀 Iniciar proceso" en cian → "🛠️ Cerrar mantenimiento" en azul (rojo si es correctivo) → "📄 Ver reporte" en gris. El sistema le dice al operador qué hacer sin que tenga que pensar.
Módulo 6
Solicitudes y tickets
Sistema de tickets para reportar novedades sobre equipos. Cualquier usuario puede crear una solicitud, pero solo el administrador puede resolverla.
⚠️
Requiere: Inventario (equipo existente)
Toda solicitud debe estar vinculada a un equipo específico. La solicitud queda en el historial del equipo y puede derivar en un mantenimiento.
🚨
Priorización automática
Las solicitudes de tipo Daño y Baja aparecen siempre al tope de la lista con borde rojo y animación de alerta. Las de tipo Anomalía e Ingreso tienen estilo neutro. Los resueltos aparecen al final con opacidad reducida.
📋
3 opciones de resolución
El admin resuelve con 3 opciones visualmente diferenciadas: Crear mantenimiento (cian — el equipo va al taller), Dar de baja (rojo — el equipo se retira definitivamente), Desestimar (gris — era falsa alarma, no se hace nada).
🔗
Vínculo con mantenimiento
Si la solicitud deriva en mantenimiento, la ficha de la solicitud muestra el enlace al mantenimiento creado (ej: MAN-2024-001). Esto permite trazar el problema desde el reporte inicial hasta el cierre técnico.
Módulo 7
Servicios prestados
Módulo de datos puros para facturación. Muestra cuántos días lleva activo cada equipo por cliente. No calcula precios ni tarifas — eso lo decide Ingemedic externamente.
ℹ️
Este módulo solo lee datos — no modifica nada
Es una vista de consulta sobre las órdenes de servicio y entregas existentes. Los datos se calculan a partir de la diferencia entre la fecha de entrega y la fecha de devolución (o la fecha actual si el servicio sigue activo).
⏱️
Días activos en tiempo real
Los servicios activos (sin devolución) tienen borde cian y badge pulsante "En campo". Los días se calculan hasta hoy. Los finalizados muestran el total de días del período.
📊
Filtros y agrupación
Filtro por estado (Activos / Finalizados / Todos), por mes, y buscador por cliente o código de equipo. Toggle para agrupar todos los equipos de un mismo cliente en un bloque.
📥
Exportar CSV
Exporta exactamente los datos del filtro activo a un archivo CSV listo para abrir en Excel. Incluye: orden, cliente, equipo, serial, fecha entrega, fecha devolución, días activos y estado.
Módulo 8
Bitácora del sistema
Registro automático e inmutable de todas las acciones del sistema. Solo el administrador tiene acceso. No requiere ninguna acción del usuario — se escribe sola.
✅
Este módulo solo lee — no se puede modificar
La bitácora es un log de auditoría. Registra automáticamente quién hizo qué, cuándo y en qué módulo. Nadie puede editar ni borrar registros de la bitácora.
📜
Registro en lenguaje natural
Cada acción se traduce a texto legible. Por ejemplo: "Carlos Díaz completó la entrega del equipo EQ-0023 para CAJACOPI" en lugar de mostrar código técnico de base de datos.
🔍
Filtros de auditoría
Filtrar por módulo (Inventario, Entregas, etc.), por tipo de acción (Crear, Editar, Cambiar estado, Sesión) y por período (Hoy, Esta semana, Todo). Buscador por usuario o descripción del evento.

Módulo 9
Configuración
Panel de control del sistema. Solo accesible para el administrador. Permite gestionar usuarios, roles, catálogos y plantillas de documentos.
👤
Gestión de usuarios
Crear, editar, activar y desactivar usuarios del sistema. Cada usuario tiene nombre, correo y rol (Administrador o Repartidor).
🔒
Roles y permisos granulares
Tabla de checkboxes que define qué puede hacer cada rol en cada módulo: Ver, Crear, Editar, Eliminar. Estos permisos se reflejan en tiempo real en lo que ve cada usuario al iniciar sesión.
🎛️
Catálogos configurables
Los estados y tipos del sistema no están hardcodeados — viven en catálogos que el admin puede editar. Estados de equipo, tipos de cliente, tipos de solicitud, estados de mantenimiento. Cada elemento tiene nombre, descripción y color de etiqueta.
📄
Plantillas de documentos
Lista de los documentos legales configurados en el sistema. El administrador puede previsualizarlos y en producción podrá editarlos con un editor de texto enriquecido (TipTap) con marcadores protegidos ({{cliente_nombre}}, {{equipo_serial}}, etc.).
🎨 Para Alejandro — layout de doble navegación
La configuración tiene su propio sidebar secundario dentro del área de contenido, con 4 secciones: Acceso, Catálogos, Sistema, y Plantillas. Esto evita saturar el menú principal con opciones que el admin solo toca ocasionalmente.

Antes de diseñar los formularios de cada módulo, tener en cuenta qué datos vienen de otros módulos y cómo se relacionan.

Clientes Ninguno Clientes disponibles para órdenes
